## 1. 핵심 프레임워크 3가지

### STAR — 가장 범용적인 구조

| 단계 | 의미 | 이 프로필 적용 |
|------|------|--------------|
| **S** Situation | 배경·계기 | 연준 정책 전환기 / POSCO 철강 팀 배정 |
| **T** Task | 당면 과제 | 50개 변수 과적합 문제 / 도메인 모르는 상황 |
| **A** Action | **직접 취한 행동** ← 가장 길게 | 3단계 추정 프레임 설계 / 공정 원리 먼저 학습 |
| **R** Result | 수치화된 결과 | R² 0.67→0.76, 100% 부호 일관성 / 팀 방향 제시 |

**A(Action)에서 흔한 실수:** "열심히 분석했다" → "LASSO로 변수 선택 후 Post-LASSO OLS로 재추정해 축소 편향을 제거했다"

---

### 3C4P — 작은 경험도 크게 만드는 구조

**3C: 경험을 객관화하는 시각**
- **Company (내가 속한 팀·조직)**: 이 상황에서 조직이 해결해야 할 문제는?
- **Competitor (비교 대상)**: 기존 방식, 다른 사람들의 접근과 무엇이 달랐나?
- **Customer (결과의 수혜자)**: 내 분석/결정이 누구에게 어떤 가치를 줬나?

**4P: 스토리텔링 흐름**
- **Picture**: 독자가 장면을 그릴 수 있는 구체적 상황
- **Promise**: "이 경험이 왜 이 직무에서 의미 있는지"를 먼저 선언
- **Prove**: 데이터와 행동으로 증명
- **Push**: 입사 후 어떻게 확장할 것인지

**예시 적용 (논문 소재):**
- Picture: "68개 데이터 포인트로 검증이 끝났을 때, 초기 결론이 틀렸다는 것을 알았습니다"
- Promise: "저는 좋아 보이는 결과일수록 더 의심합니다"
- Prove: Baa 스프레드 불안정, Ret_Corp_InvGrade 73% 선택 빈도 / 100% 부호 일관성
- Push: "이 검증 원칙을 실제 의사결정 모델에도 동일하게 적용하겠습니다"

---

### 필살기 — 면접관이 기억하는 단 하나의 장면

**정의:** 직무 적합성을 가장 잘 보여주는, 반복해서 쓸 수 있는 핵심 경험 1개

**필살기의 조건:**
1. 직무 요구사항과 직접 연결됨
2. 구체적인 행동과 수치가 있음
3. 본인만 할 수 있는 고유한 스토리
4. 한 문장으로 요약 가능

**이 프로필의 필살기 후보 2개:**

> **후보 A — "68회 반복 검증"**
> "좋아 보이는 초기 결과를 그대로 믿지 않고, 100거래일 윈도우를 68회 반복해서 검증한 끝에 처음과 다른 결론에 도달했습니다."
> → 검증 집착, 연구 태도, 통계 엄밀함을 한 번에 증명

> **후보 B — "WSL 프로토타입으로 팀 설득"**
> "말로 설득하는 대신, 그날 안에 WSL에서 작동하는 프로토타입을 만들어 팀에 보여줬습니다."
> → 실행력, 커뮤니케이션, 판단력을 한 번에 증명

**직무별 사용 전략:**

| 포지션 | 사용할 필살기 |
|--------|-------------|
| Macro DS, Quant Researcher | 후보 A (검증 집착) |
| Decision Scientist, 핀테크 DS | 후보 B (실행력) → A (검증)으로 심화 |
| AI Risk Engineer | 후보 A + 백테스팅 버그 발견 에피소드 |

## 2. 수치화 전략

**기본 원칙:** 결과 수치 > 행동 수치 > 규모 수치

| 수치 유형 | 나쁜 예 | 좋은 예 |
|----------|--------|--------|
| 결과 수치 | "분석 정확도가 올랐다" | "R² 0.67 → 0.76 향상" |
| 행동 수치 | "여러 번 검증했다" | "68회 반복 추정" |
| 규모 수치 | "많은 변수를 분석했다" | "50개 거시경제 변수" |
| 검증 수치 | "안정적인 결과가 나왔다" | "73% 선택 빈도, 100% 부호 일관성" |

**이 프로필의 사용 가능한 수치 목록:**

| 소스 | 수치 |
|------|------|
| 논문 | 50개 변수, 68회 반복, R² 0.67→0.76, 73% 선택빈도, 100% 부호 일관성 |
| mlb-stats | 4,940+ lines, 180 tests passing, wOBA/wRC+/FIP 7개 지표 직접 구현 |
| market_density | 5단계 파이프라인, VaR/ES/옵션가격, NLL/CRPS/PIT 3종 검증 |
| 야구 | 182,449 PA, 오타니 .310/.968/.1.093 OPS |
| EIMAS (참고) | 7 에이전트, 4개 레짐, 신뢰도 65~75% |

## 3. 항목별 작성 전략

### 지원동기

**구조:** 만남의 계기 → 이 회사의 특별함 (경쟁사와 차별점) → 내 경험이 왜 맞는지 → 입사 후 비전

**피해야 할 표현:**
- "업계 1위여서", "성장 가능성이 높아서"
- "AI에 관심이 많아서"

**써야 할 표현:**
- 회사의 특정 제품/기술/사업 방향을 명시
- "저는 [구체적 경험]을 통해 [역량]을 키웠고, 귀사의 [구체적 과제]에 이를 적용하고 싶습니다"

---

### 직무역량

**구조:** STAR + 수치화 + "이 경험이 이 직무에서 어떻게 쓰이는가"

**금융AI 직군 특수 포인트:**
- 기술 스택을 나열하지 말고, **어떤 문제를 어떤 기술로 해결했는지** 서술
- 나쁜 예: "Python, LASSO, statsmodels 사용 가능"
- 좋은 예: "금융 시계열의 자기상관 문제를 해결하기 위해 Newey-West HAC 표준오차를 적용했고, 그 결과 추정 신뢰구간이 실질적으로 달라졌습니다"

---

### 성장경험 / 실패경험

**핵심:** 실패나 어려움 자체가 아닌, **그 이후의 행동과 원칙 변화**가 포인트

**이 프로필의 강력한 소재:**
> 백테스팅에서 4년 누적 수익률 8360%가 나왔을 때 — 의심했다.
> 숏 포지션 수수료 누락 버그를 발견했다.
> 이후: 모든 파이프라인에 데이터 무결성 체크를 선행 단계로 넣는 것을 기본 원칙으로 삼았다.

→ "좋아 보이는 결과를 의심하는 원칙"이 어떻게 형성됐는지를 보여주는 완벽한 소재

---

### 입사 후 포부

**구조:** 단기 (1-2년) → 중기 (3-5년) / 추상적 포부 금지

**피해야 할 표현:** "열심히 배우겠습니다", "회사 발전에 기여하겠습니다"

**써야 할 표현:**
- "현재 [구체적 갭]이 있는데, [구체적 방법]으로 채우겠습니다"
- "입사 후 [구체적 역할]에서 [구체적 가치]를 만들겠습니다"

## 4. 금융AI 직군 특화 팁

### 이 직군 채용담당자가 가장 많이 보는 것

1. **"이 사람이 실제로 코드를 짰는가?"** → GitHub 링크, 코드 수치, 테스트 수
2. **"도메인을 아는가, 기술만 아는가?"** → 금융 개념을 설명에 자연스럽게 사용
3. **"결과를 의심할 줄 아는가?"** → 좋은 결과에서 멈추지 않은 사례
4. **"혼자 설계했는가, 따라 했는가?"** → "직접 설계했다"는 표현과 근거

### 차별화 포인트 (대부분 지원자와의 차이)

| 대부분 지원자 | 이 프로필 |
|-------------|---------|
| ML 라이브러리 사용법 학습 | 방법론을 직접 설계 (3단계 추정 프레임워크) |
| 모델 만들고 끝 | 모델 검증 (68회 롤링, 버그 발견) |
| Python 할 줄 안다 | 4,940줄 / 180 tests / asyncio / FastAPI |
| 금융 관심 있다 | 경제학 전공 + 연준 정책 논문 + EIMAS |

## 5. 제출 전 체크리스트

- [ ] 회사명·직무명 정확히 기재됐는가
- [ ] 수치가 최소 항목당 2개 이상 있는가
- [ ] "열심히", "최선을", "관심이 많아" 없는가
- [ ] A(Action)이 가장 긴 문단인가 (STAR 기준)
- [ ] 소리 내어 읽었을 때 자연스러운가
- [ ] 필살기 에피소드가 1개 이상 들어갔는가
- [ ] GitHub 링크 또는 프로젝트 언급이 있는가
- [ ] 회사의 구체적인 제품/서비스/기술을 언급했는가
